# Sprint Build

복잡한 작업을 작은 단위로 나누어 기획→구현→평가 사이클을 반복합니다.
각 Phase는 사용자 checkpoint 후 다음 Phase로 진행합니다.

## Phase 1: Spec & Contract

작업 유형을 감지하고 완료 기준(Definition of Done)을 정의합니다.

### 1-0. 준비 상태 확인 (DOR)

작업에 필요한 정보가 충분한지 확인합니다.

- 관련 파일의 전체 코드를 읽었는가?
- 사용 중인 라이브러리/API의 버전과 스펙을 파악했는가?
- 기존 코드 패턴과 제약사항을 이해했는가?

부족한 항목이 있으면 사용자에게 구체적으로 요청한 후 다음 단계로 진행합니다.

### 1-1. 작업 유형 감지

| 유형      | Phase 1 초점                 |
| --------- | ---------------------------- |
| 기능 구현 | 사용자 스토리 → 완료 기준    |
| 버그 수정 | 재현 조건 → 해결 기준        |
| 리팩토링  | 현재 상태의 문제 → 개선 기준 |

### 1-2. 완료 기준(DOD) 작성

구체적이고 검증 가능한 체크리스트 형태로 작성:

- 핵심 동작/기능 요구사항
- 경계 조건 및 엣지 케이스
- 성능/접근성 기준 (해당 시)

### 1-3. 사용자 checkpoint

DOD를 사용자에게 제시하고 승인을 받습니다.
사용자가 수정하면 DOD를 업데이트합니다.

DOD 승인 후, 사용자에게 질문합니다:
"이번 스프린트의 구현 계획을 interactive planning으로 수립할까요?
 (각 설계 선택지를 논의하며 진행합니다)"

- **예**: `plan-interactively` 스킬을 호출하여 Phase 2(Implement) 전에 계획 수립
- **아니요**: 기존대로 agent가 자율적으로 Phase 2로 진행

---

## Phase 2: Implement

DOD에 맞춰 코드를 작성합니다.

### 2-1. 스프린트 분할

작업이 클 경우 기능 단위로 분할합니다.
각 스프린트는 독립적으로 검증 가능해야 합니다.

### 2-2. 코드 작성

- 기존 프로젝트 규칙(AGENTS.md, docs/agents/rules/*)을 준수
- 관련 스킬 활용:
  - UI 작업 → `frontend-design` 스킬 참조
  - React 코드 → `react-best-practices` 스킬 참조
- 기존 코드 패턴을 따르고, 불필요한 추상화 지양

### 2-3. 자가 검증

- 린트/컴파일 통과 확인
- 기존 테스트 깨짐 없는지 확인
- 스프린트 단위 기본 동작 확인

---

## Phase 3: Evaluate

DOD 대비 엄격하게 평가합니다.

### 3-1. 기본 평가 기준 (모든 작업 유형)

1. **정확성**: DOD의 각 항목이 실제로 충족되었는가?
2. **완성도**: 엣지 케이스, 에러 상황이 처리되었는가?
3. **코드 품질**: 기존 코드와 일관성, 불필요한 복잡도가 없는가?

### 3-2. 작업 유형별 추가 기준

- **기능 구현**: 사용자 경험, 예상치 못한 동작 여부
- **버그 수정**: 근원 원인 제거 여부, 유사 버그 존재 여부
- **리팩토링**: 동작 변경 없음, 성능 퇴행 없음

### 3-3. 평가 원칙

- "문제없음"이라고 관대하게 평가하지 않습니다.
- 실제 사용 시 발생할 수 있는 작은 디테일 누락까지 잡아냅니다.
- 피드백은 구체적이고 재현 가능해야 합니다
  (예: "X 버튼 클릭 시 에러 발생" O / "버그 있음" X)

### 3-4. 결과 분기

- **통과**: 스프린트 완료, 다음 스프린트 또는 Phase 종료
- **사소한 이슈**: → Phase 4(수정) → Phase 3(재평가)
- **근본적 문제(스펙 오류 등)**: → Phase 1로 복귀하여 DOD 재정의

---

## Phase 4: Fix & Iterate

Phase 3의 피드백을 바탕으로 수정합니다.

### 4-0. 수정 실행

- 피드백 항목별로 수정 사항을 명확히 정리
- 수정 후 Phase 3로 복귀하여 재평가
- 기준 충족까지 반복 (사용자 판단으로 조기 종료 가능)

### 4-1. 반복 한계 (Escape Hatch)

동일한 문제로 3회 이상 수정이 실패할 경우:
- 반복을 중단하고 사용자에게 상황을 보고합니다.
- 실패 원인과 사용자 개입이 필요한 사항을 명확히 제시합니다.
- 방향성 수정 또는 작업 범위 재조정을 요청합니다.

---

## 주의사항

- **기획 단계**: 기술적 디테일을 과도하게 사전 결정하지 않습니다.
  실수가 전체 구현에 퍼지는 것을 방지하기 위해, 구체적인 How는 구현 단계에서 결정합니다.
- **평가 단계**: 엄격하게. "문제없음" 관대 평가 금지.
- **피드백**: 구체적이고 실행 가능한 피드백만 제공합니다.
- **DOD 준수**: Phase 1에서 합의한 기준이 곧 완료 판단 기준입니다.
  중간에 기준을 임의로 낮추거나 변경하지 않습니다.
