4-layer 구조

L1. Universal source platform 중립

모든 룰 본문 1곳. agent 가 trigger 시점에 read.
  • 📁 memory/ — 룰 (workflow / ux / runbook / conventions)
  • 📁 docs/ — sprint artifact
  • 📁 scripts/check-*.sh — bash, universal
  • 📄 lefthook.yml — git hook, universal

L2. Universal entry agent 공통

모든 agent 의 공통 시작점. 작업 type → memory path 매트릭스.
  • 📄 AGENTS.md (신설) — 5-10줄, 작업 type 매트릭스
  • 📄 CLAUDE.md — Claude 가 자동 read (현재 60줄 → 1-3줄 redirect 권장)
  • 📄 codex / cursor 의 entry — 같은 본문 또는 1줄 redirect

L3. Per-agent wrapper platform 별 thin

3-5줄. role 명시 + memory path read 강제. 본문 0 embed.
  • 📁 .claude/agents/<role>.md — Claude format
  • 📁 .codex/agents/<role>.<ext> — Codex format (TBD)
  • 📁 .cursor/... — Cursor format (TBD)
  • 본문 = role + read 강제 path 1-2개

L4. Hook wiring script universal + 형식 platform

Script 본체는 universal. 각 platform 의 hook config 가 호출.
  • 📄 scripts/hooks/check-god-file.sh (universal)
  • 📄 .claude/settings.json PostToolUse → script 호출
  • 📄 .codex/config → 같은 script 호출 (TBD)
  • 📄 lefthook.yml pre-commit / pre-push → universal git layer

Lazy 의 단점 — "agent 가 read 안 하면?"



  
가장 가벼운 형태의 위험 = agent 가 wrapper 만 보고 memory path read 안 한 채 작업 시작. 해결:

  
    
    
  • Wrapper 의 강제 명령 — "read 후 진행. read 없이 작업 = 룰 위반" 명시. agent 의 alignment 에 의존.
    • 
    
  • Universal entry 의 enforcement 기준 — AGENTS.md 가 "memory path 인용 없이 작업 = 룰 위반" 명시.
    • 
    
  • PostToolUse hook 검증 (옵션, deferred) — agent 의 첫 Edit/Write 직전 memory path read tool call 있었는지 검사. 복잡 — R2 deferred 영역.
    •
