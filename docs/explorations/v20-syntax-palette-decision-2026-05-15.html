V20 — Syntax palette source-of-truth

ADR 0023 (Sprint 257, generate-syntax-palette.ts) vs 새 시방서의 1728 수동 값

핵심 질문

새 시방서가 박은 72 테마 × 2 모드 × 12 토큰 = 1728 값 과 ADR 0023 의 "per-theme primary 에서 HSL 도출" 원리 중, 어느 쪽을 source of truth 로 삼을 것인가? 이 결정이 미래 76번째 테마 추가, lamborghini 같은 의문점 해결, drift 위험 모두를 결정한다.

옵션 A

(a) 스크립트 폐기

수동 데이터 → source of truth

접근

  • generate-syntax-palette.ts 삭제
  • 새 시방서 1728 값을 themes.css 에 그대로 append
  • ADR 0023 → Superseded
  • 신규 ADR: "syntax palette 는 시방서 산출물 manual import"

장점

  • 당장 작업량 0
  • 디자이너 의도 (lamborghini 등 brand 예외) 100% 보존
  • 결정 룰 추가 없음

단점

  • 76번째 테마 추가 시 사람이 12 값 입력 (forget 사고)
  • 1728 값 사람 관리 — drift 위험
  • "lamborghini light = cool_dark" 의문이 영원히 미해결
  • ADR 0023 정신 폐기 (감점)
1년 후 시나리오: 76번째 테마 "tableplus" 추가. 개발자가 themes.css 에 base 토큰만 추가하고 syntax 12 값 잊음 → 그 테마에서 syntax highlight 가 빈 색 (transparent) 으로 떨어짐. PR review 에서 우연히 발견 또는 production 도달.
추천 옵션 B

(b) 스크립트 12 토큰 확장

자동 도출 → source of truth

접근

  • generate-syntax-palette.ts 의 HSL 도출 룰을 3 → 12 토큰으로 확장
  • 신규 ADR 0026 — ADR 0023 의 Extended (Superseded 아님)
  • 본 시방서 1728 값은 검증 베이스라인: 스크립트 재생성 결과와 diff
  • diff 가 크면 룰 또는 시방서 중 어느 쪽이 맞는지 토론

장점

  • 미래 테마 자동 cover — 76·77·78… 무한 확장
  • per-theme 일관성 보장 (사람 손실수 0)
  • ADR 0023 정신 보존 (Extended)
  • drift 위험 0 — 재생성으로 reset
  • 새 시방서 데이터를 1회 검증에 활용 → lamborghini 의문 해결

단점

  • 9 토큰 (atom/builtin/property/punct/operator/function/comment/type/error) HSL 룰 정의 비용
  • brand 예외 (lamborghini light=dark) 는 룰에 박아야
  • 스크립트 코드량 +200줄
1년 후 시나리오: 76번째 테마 "tableplus" 추가. 개발자가 base 토큰만 추가, pnpm tsx scripts/generate-syntax-palette.ts 실행 → 12 토큰 자동 계산되어 themes.css 에 추가. 0 사람 손길. lamborghini 류 brand 예외도 룰에 박혀 있어 자동 적용.
옵션 C

(c) 혼합

자동 9 + 수동 3 (분리된 책임)

접근

  • "규칙적" 9 토큰 (keyword/string/number/function/operator/type/property/comment/error) → 자동 확장
  • "비규칙적" 3 토큰 (atom/builtin/punct) → 수동 입력
  • 두 source 가 같은 파일 안에서 공존

장점

  • 규칙성 있는 9 토큰은 자동 cover
  • 디자이너 의도 살릴 3 토큰은 수동 (유연성)

단점

  • 책임 라인 모호 — 어느 토큰이 자동인지 매번 확인
  • 스크립트 가 themes.css 의 일부 라인만 overwrite 하는 fragile 로직
  • 76번째 테마 시 자동 9 + 수동 3 — forget 사고 still 가능
  • 두 source 의 일관성 보장 메커니즘 추가 필요
1년 후 시나리오: 새 테마 추가 시 개발자가 "atom/builtin/punct 는 자동인가 수동인가?" 매번 확인. 문서 의존 → 문서가 stale 되면 forget 사고. b 보다 worst case 가 같고, best case 만 조금 다름.

추천: (b) 확장

이유 — TablePlus parity 가 목표인 본 프로젝트는 테마가 76→100+ 확장 가능성 있음. 사람이 12 값 × 100+ 테마 = 1200+ 값 관리는 비현실적. ADR 0023 의 "primary 에서 도출" 정신을 보존하면서 12 토큰까지 cover 하면 본 시방서 1728 값은 1회 검증 후 자동 산출물로 교체 — 디자이너 노력은 룰 정의로 흡수.

"lamborghini light = cool_dark" 같은 의문 케이스도 brandExceptions: { lamborghini: { light: 'cool_dark' } } 같이 룰에 명시되면 의도가 코드에 박힌다.
