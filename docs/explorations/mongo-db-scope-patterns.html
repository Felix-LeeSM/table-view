Mongo "현재 DB" UX 패턴 비교

Mongo 의 database 가 PG schema 처럼 cross-query 자유라서, "어디서 동작 중인가" 의 시각화가 도구마다 갈린다. 5개 도구 패턴 + 이 프로젝트의 추천안. 각 mock 의 노랑 chip = global state, 파랑 chip = tab-local state.

읽는 법

기호 의미
global connection 전체가 공유하는 state. 한 곳에서 바꾸면 모든 tab 영향.
tab tab 마다 별도 state. 한 tab 의 DB 변경이 다른 tab 에 영향 없음.
selected in sidebar 사용자가 마지막으로 클릭한 위치.

💡 Sidebar = 컬렉션 browser. 쿼리는 grid 안 filter bar 에서만 작성. free-form mongosh 은 별도 패널이고, 거기서만 use 가 동작. 두 mental model 이 완전히 분리되어 redundancy 가 없음.

장점

  • "현재 DB" 가 두 군데 같은 게 없다 — sidebar 는 explore, REPL 은 stateful execute.
  • mongosh 의 REPL semantics 가 그대로 살아남음.

단점

  • 이 프로젝트는 mongosh 를 query tab 의 메인으로 박았음 — REPL 패널 분리 = sprint A3 reverse.
  • 두 surface 가 따로 있다는 걸 사용자가 배워야 함.

💡 Tab 안 작은 db = app ▾ chip 이 tab 단위 state. sidebar 와 비동기화 — sidebar 클릭이 실수로 실행 context 를 흔드는 사고 방지. sidebar 우클릭 → "New query here" 가 sidebar→tab 의 유일한 명시적 다리.

장점

  • tab 갯수 안 늘리고 잠깐 DB 전환 가능.
  • "어떤 query 가 어디서 도느냐" 가 tab 안에서 self-contained.
  • backend 의 stateless 한 IPC signature 와 1:1 정합 (이미 wrapper 가 `(connId, db, coll, …)` 인자).
  • RDB / Mongo 가 같은 mental model 로 통일 (DataGrip 이 둘 다 처리하는 방식).

단점

  • "global active DB" 라는 개념이 사라져서, 처음 connect 직후 어디 있는지 학습 필요.
  • tab chip 의 popover UX 를 잘 만들어야 함 (작지만 자주 쓰는 위젯).

단점

  • Mongo 의 stateless 한 IPC 와 mismatch.
  • 다중 tab 이 다른 DB 동시 보기 어려움.

• orders

💡 PG 의 강 격리 모델을 그대로 노출. database = connection. 이 프로젝트 의 RDB 쪽은 sub-pool 로 부드럽게 처리하는데 (active sub-pool 전환), Mongo 에는 어울리지 않음 — Mongo 는 강 격리가 아님.

장점

  • PG semantics 와 1:1 정합 (RDB 에서는 자연스러움).

단점

  • Mongo 의 cross-DB 자유와 충돌.
  • 다중 DB 작업 = 다중 connection — Mongo 사용자에게 무거움.

결론 — 이 프로젝트의 추천 (패턴 3)

  1. Toolbar global DbSwitcher 는 Mongo paradigm 에서 제거. RDB 는 그대로 유지 (PG database 강 격리 때문).
  2. Query tab 의 좌상단에 inline DB chip — db = app ▾. 클릭 → popover 로 DB 선택. tab-local state.
  3. Sidebar 와 tab chip 은 비동기화. sidebar 클릭은 explore 만, tab 의 db scope 안 흔든다.
  4. sidebar → tab 다리는 명시적 액션 1개 — sidebar database row 우클릭 → "New query here" 가 prefilled tab.
  5. 새 tab 의 default db = connection 의 default_db 필드 (모델에 이미 있음). sticky last-touched 같은 추가 state 안 만든다.
  6. backend switch_active_db / current_active_db 는 dead code — Mongo 쪽 별도 후속 sprint 에서 제거 (RDB 는 유지).

DataGrip 패턴이 가장 가까운 이유: (a) Mongo + RDB 모두 처리하는 도구의 검증된 UX, (b) backend 가 이미 stateless 한 wrapper 표면 (`(connId, db, coll, …)`) 과 1:1 정합, (c) "한 view = 한 state" 원칙.
