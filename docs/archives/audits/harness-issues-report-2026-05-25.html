기반 Foundation

나머지 모든 문제의 근원. 이 영역이 해결되면 아래 문제들의 강도가 크게 약화됩니다.

#8
용어가 정렬되어 있지 않다

작업 지시 시 "윈도우", "패널", "뷰", "사이드바" 같은 용어가 사용자, memory, 코드베이스 사이에서 일관되지 않습니다. 같은 것을 다른 이름으로 부르거나, 같은 이름으로 다른 것을 부르면 지시가 복잡해지고 의도가 분기합니다.

Agent는 매 세션 새로 시작하므로 암묵적 합의가 작동하지 않습니다. 명시적 용어 사전 없이는, sprint마다 다른 용어를 사용하고, 이게 코드와 문서에 반영되어 전체 일관성이 무너집니다.

  • 사용자가 "사이드바"라고 하면 agent는 무엇을 이해하는가?
  • 코드에서는 ConnectionPanel, memory에서는 "connection navigator", UI에서는 "sidebar" — 어느 것이 정답인가?
Foundation High impact
연결: #9 코드 패턴 불일치, #12 Unknown unknowns, #2 규칙 준수
#17
일반적인 코딩 원칙이 문서화되어 있지 않다

언제 hook을 쓰고 언제 utility를 쓸지, 에러 처리는 어떻게 할지, 함수는 어디까지 쪼갤지 같은 기본 원칙이 memory에 체계적으로 정리되어 있지 않습니다. Domain-specific 규칙(Rust 컨벤션, React 컨벤션)은 있는데, 그 밑에 깔려야 할 일반 원칙이 빠져 있습니다.

Agent가 매번 상황에 따라 임의로 결정하니, sprint마다 다른 철학이 반영됩니다.

Foundation High impact
연결: #9 코드 패턴 불일치의 뿌리, #2 규칙 준수의 전제
#18
Import된 skill과 custom harness가 엉켜 있다

Grill, TDD, delivery 등은 다른 repository에서 가져온 것이고, harness는 직접 만든 것입니다. 설계 배경과 가정이 다르니 서로 잘 엮이지 않습니다.

  • Grill은 단일 질문에 집중하는 인터뷰 모델
  • Harness는 multi-sprint pipeline
  • 두 개가 만나야 할 지점(설계 결정 → sprint contract)이 명확하지 않음

작업 방식이 하나로 정형화되어야 1-23번 문제를 아무리 고쳐도 다시 흩어지지 않습니다.

Foundation High impact
연결: 모든 skill/agent 문제의 상위, #22 라우팅의 전제
#22
자연어 지시가 적절한 subagent로 자동 라우팅되지 않는다

"리뷰해라"라고 하면, 알아서 적절한 컨텍스트를 가진 subagent가 spawn되어야 합니다. 지금은 사용자가 어떤 skill을 쓸지, 어떤 agent를 부를지 알아야 합니다. Intent → agent 매핑이 수동입니다.

사람이 orchestrator 역할을 직접 수행해야 하니, 자율 개발의 목표와 멀어집니다.

연결: #18 통합, #14 관찰 불가

인지 Cognition

Agent가 세계를 이해하고 판단하는 방식의 한계.

#1
Agent가 큰 그림을 못 본다

Sprint 단위로 쪼개진 contract만 받으니, 자기가 어디를 건드리고 있는지 시스템 전체 맥락에서 이해하지 못합니다. 국소적으로는 올바른 결정이 전체적으로는 잘못된 결정일 수 있습니다.

Cognition High impact
연결: #10 cross-sprint 회귀, #12 unknown unknowns
#2
규칙 준수가 보장되지 않는다

Memory, convention, skill 사용 지시가 전부 prompt의 soft requirement입니다. "이렇게 하세요"라고 적어놓았을 뿐, 실제로 따랐는지 검증하는 수단이 없습니다. Agent가 규칙을 무시하거나 누락해도 발견이 안 됩니다.

Cognition High impact
연결: #17 원칙 부재가 원인, #9 #11 로 파급
#12
Agent가 모르는 것을 모른다

명시적으로 주어진 정보(contract, memory, 코드)는 처리할 수 있지만, 주어지지 않은 정보의 부재를 인식하지 못합니다.

  • Security consideration이 contract에 없으면 고려하지 않고, 누락이라는 사실도 모름
  • Performance, accessibility, backward compatibility도 마찬가지

이건 LLM의 구조적 한계에 가깝습니다. 외부에서 질문을 던지는 역할이 필요합니다.

Cognition Structural limit
연결: #8 용어가 정렬되면 일부 완화, #1 큰 그림 문제와 공유
#16
Caveman 모드가 subagent와 compaction 후에 유지되지 않는다

Caveman 스킬은 모든 agent의 기본 모드여야 하는데, subagent spawn 시 상속되는지 보장이 없습니다. 특히 Codex는 context compaction 후 caveman 지시를 잃어버립니다.

결과적으로 subagent가 장황한 출력을 내면, 그게 상위 agent의 context를 다시 잠식합니다. Caveman은 token 절약이 목적인데, 미적용 시 token 낭비가 증폭되는 구조입니다.

Cognition Mid impact
연결: #6 context 오염과 직결, 악순환 구조

프로세스 Process

작업이 실행되고 추적되는 방식의 한계.

#3
시나리오-테스트 연결고리가 없다

memory/conventions/testing-scenarios/에 시나리오가 정의되어 있지만, 실제 테스트 코드와의 대응 관계를 알 수 없습니다. "시나리오 X를 커버하는 테스트가 존재하는가?"라는 질문에 답할 수 없습니다.

#4
작업 후 문서화가 안 된다

Sprint가 끝나면 handoff.md를 채워야 하고, 학습한 내용을 memory에 옮겨 적어야 합니다. 하지만 이게 제대로 이루어지지 않습니다. 템플릿이 있어도 내용이 대충 채워지거나 아예 건너뛰어집니다.

#7
Harness가 실제로 동작했는지 모른다

Sprint가 PASS로 끝났다고 해서 harness의 전체 workflow가 제대로 순차적으로 실행되었는지 알 수 없습니다. Phase를 건너뛰었는지, Planner가 실제로 코드베이스를 읽었는지, Contract가 sprint scope를 정확히 반영했는지 — 어느 것도 외부에서 검증할 수 없습니다.

결과물(코드)만 보이고 과정은 불투명합니다.

연결: #2 준수 보장, #10 회귀, #13 worktree
#13
Worktree 격리 패턴이 지켜지는지 모른다

Main agent를 orchestrator로, worktree에서 작업을 진행하는 구조를 설계했습니다. 하지만 이 패턴이 실제로 지켜지는지, 작업이 main 브랜치를 오염시키지 않았는지 확인할 수 없습니다.

check-main-worktree-source-edit.sh hook이 있긴 하지만, 이건 편집 정책이지 workflow compliance가 아닙니다.

#15
미리 만든 sprint 문서가 오히려 흐름을 끊는다

Sprint 문서를 사전 작성 후 작업을 진행해 봤지만, 상황이 변하면 문서와 실제 작업이 어긋나고, 문서에 맞추려다 자연스러운 개발 흐름이 깨집니다. 구조가 너무 강하면 적응력이 떨어집니다.

구조화와 유연성의 딜레마: 구조가 없으면 agent가 방황하고(1-12), 구조가 너무 강하면 사람이 흐름을 잃습니다.

#20
작업 추적이 repo 내부 파일에만 있다

Sprint가 docs/sprints/ 디렉토리에 파일로만 존재합니다. 진행 상황, 계획, 완료 여부를 repo 안에서만 알 수 있고, 외부에서는 보이지 않습니다. GitHub Issue, Milestone, Project board 같은 추적 수단과 연결이 없습니다.

잠재적 해결책이기도 합니다: GitHub 연결로 #7 #14 #15 #19를 완화할 수 있습니다.

#23
Worktree ref 관리 hook이 효율적으로 동작하는지 모른다

FETCH_HEAD, ORIG_HEAD 등 git 내부 ref와 worktree의 충돌을 방지하기 위해 hook을 작성했지만, 이 hook들이 실제로 효율적으로 동작하는지 검증이 필요합니다.

문제가 발생했을 때 대응식으로 작성된 패치들이라, 근본 설계가 올바른지 재검토가 필요합니다. Git 내부 FETCH_REF 관리와 충돌 문제가 근원입니다.

결과 Outcome

최종적으로 나타나는 증상. 기반·인지·프로세스 문제가 겹쳐서 발생합니다.

#5
Memory가 방대해지는데 관리 체계가 없다

Memory는 계속 쌓이기만 합니다. 어떤 memory가 여전히 유효한지, 어떤 것이 구식인지, 무엇이 중요하고 무엇이 잡음인지 판단하는 메커니즘이 없습니다.

200줄 cap은 파일 하나의 크기 제한일 뿐, 전체 memory 생태계의 건강성을 보장하지 못합니다.

Outcome High impact
연결: #4 문서화 누락, #11 피드백 루프
#6
검증을 위한 enforcement가 context를 오염시킨다

문제들을 해결하려고 hook으로 검사하고 경고를 넣으면, agent의 한정된 context window를 검증 메시지가 잠식합니다. 핵심 작업 정보보다 경고 메시지가 더 많아지는 역설이 발생합니다.

해결 방향: push(agent에게 알림) 대신 pull(Evaluator에게만 전달). Generator는 코드에 집중, Evaluator가 side-channel compliance 데이터를 평가에 반영.

Outcome High impact
연결: #16 caveman 미유지와 악순환, #2 준수 검증의 부작용
#9
코드 패턴 일관성이 강제되지 않는다

같은 유형의 컴포넌트를 만들 때마다 다른 패턴이 사용됩니다. 한 sprint에서는 custom hook, 다음 sprint에서는 인라인. Formatting(prettier, cargo fmt)은 잡지만, 구조적 패턴은 잡지 못합니다.

Outcome Mid impact
연결: #8 용어, #17 원칙 부재의 코드적 표현
#10
Sprint 간 회귀가 감지되지 않는다

Sprint 3이 통과해도, Sprint 1의 결과를 망가뜨렸는지 아무도 검사하지 않습니다. Evaluator는 "이 sprint의 contract"만 평가합니다. memory/lessons/에 교훈이 있지만 이미 학습한 특정 사례에 대한 방어일 뿐, 일반적인 cross-sprint 회귀 감지 체계가 아닙니다.

Sprint가 쌓일수록 누적 위험은 단조증가합니다.

Outcome High impact
연결: #1 큰 그림, #7 실행 검증
#11
피드백 루프가 닫히지 않는다

Sprint에서 실수 → Evaluator 지적 → Generator 수정은 잘 됩니다. 하지만 그 실수의 원인이 다음 sprint나 다음 feature에서 반복 방지되는지는 보장되지 않습니다.

memory/lessons/에 기록되더라도, 다음 agent 세션이 그걸 읽고 실제로 반영했는지 검증이 없습니다. 교훈이 write-only가 됩니다.

Outcome High impact
연결: #5 memory 관리, #4 문서화
#14
사용자가 작업 현황을 관찰할 수 없다

Worktree에서 작업이 진행되면 사용자는 그 안에서 무슨 일이 일어나는지 볼 수 없습니다. 진행 상황, 중간 결과, 문제 발생 — 전부 블랙박스입니다. 작업이 끝나고 PR이 올라와야 처음으로 결과를 봅니다.

사람이 감시하려 해도 감시 창구가 없고, 알아서 잘 돌기를 바랄 수밖에 없는데 그게 안 됩니다.

Outcome High impact
연결: #13 worktree, #20 추적, #7 실행 검증
#19
Human-readable 문서가 방치된다

README, plan, roadmap, features 등 사람이 읽는 문서는 memory/와는 다른 영역입니다. Agent는 memory/를 관리하도록 되어 있지만, repo 루트의 human-facing 문서는 sprint contract, hook, evaluator rubric 어디에도 포함되지 않습니다.

코드는 바뀌어도 README는 예전 그대로, roadmap은 구 시안, features는 실제 구현과 불일치합니다.

Outcome Mid impact
연결: #4 문서화 누락과 대상 다름, #20 GitHub 연결로 완화 가능
#21
Smoke test와 fixture가 관리되지 않는다

테스트 코드는 작성하지만, 그 테스트가 의존하는 fixture, mock 데이터, smoke test 인프라는 방치됩니다. DX(Developer Experience)가 고려되지 않아서, 테스트가 깨지면 고치는 비용이 초기 작성 비용보다 큽니다.

Agent는 "테스트를 작성하라"에는 반응하지만, "테스트가 지속적으로 유용하려면 무엇이 필요한가"에는 답하지 못합니다.

Outcome Mid impact
