## Goal

Build a realistic fixture/data workflow for Table View that supports:

- automated e2e checks
- daily development and QA
- edge case data verification
- performance testing
- PostgreSQL, MySQL, and MongoDB targets
- connection ordering and TablePlus-style connection import/export

The fixture system should help develop and test the database client itself, not become a separate data-generation product.

## Decisions

### Profiles

Use two named profiles:

| Profile | Size | Use |
| --- | ---: | --- |
| `e2e` | 1K-3K rows | Automated tests and remote smoke checks |
| `development` | 5K-20K rows, 10-15% multilingual/edge mix | Daily development and QA |

Reason:

- `e2e` and `development` are the two profiles likely to be run frequently. Edge-case and performance variants are deferred until the compiler proves the workflow.
- Edge-case data (NULL, encoding, escaping, unusual values) is folded into `development` at 10-15% mix instead of split into a separate profile — having to switch profiles to exercise edge data was friction without payoff.
- Performance-scale data is deferred; current product surface does not yet benchmark against 750K+ rows.

### Database Names

Use one database per profile:

```text
table_view_e2e
table_view_development
```

`table_view_test` (existing, used by cargo integration tests) stays untouched in the same container.

Reason:

- Keeps test/dev data isolated.
- Avoids accidental mutation of the e2e baseline while developing.
- Makes reset/seed behavior profile-scoped and predictable.

### Container Topology

All profile databases live in the **same container per engine**:

- One `postgres:16-alpine` container holds `table_view_test` (existing), `table_view_e2e`, `table_view_development`.
- One `mongo:7` container holds the corresponding databases by name.
- When MySQL adapter lands, the same pattern applies — one `mysql` container, one database per profile, with table prefixes (per `MySQL Mapping`).

Reason:

- Multiple databases inside a single PG / Mongo / MySQL instance is a first-class operational pattern; running one container per profile triples docker overhead and host-port management for no isolation benefit.
- Existing `docker-compose.yml` already provisions one container each — DB topology is purely a fixture-spec decision, not a docker-compose change.
- Profile-scoped reset (`reset` drops + recreates only the named DB) gives sufficient isolation; the shared volume / process is not a hazard for this workload.

### Volume Strategy

Both engines persist data across container restarts:

- `postgres` keeps the existing `pgdata` volume.
- `mongo` gains a new `mongodata` volume (currently ephemeral); add a `volumes: - mongodata:/data/db` mount and the matching top-level entry in `docker-compose.yml`.

Profile-level reset semantics are unchanged: `reset <profile>` drops and recreates the named database within the shared volume. The volume itself is never wiped by the fixture CLI — `docker compose down -v` remains the explicit "blow everything away" affordance for the user.

Reason:

- Daily dev expects yesterday's seed to still be there on `docker compose up` today. With mongo ephemeral, every restart wipes the document data and the silent-skip behaviour of `seed` (`already seeded` → noop) becomes meaningless.
- e2e is unaffected because each e2e run starts with `reset`, which drops + recreates the DB regardless of volume persistence.
- Mental-model consistency: PG persists, so Mongo should persist too — partial persistence breaks user expectation.

### Source Of Truth

Commit only a YAML fixture spec. Do not commit raw `.sql` or `.sql.gz` fixture dumps.

Reason:

- Raw SQL is target-specific and hard to reuse.
- YAML can describe logical domains, entities, relationships, sizes, and data traits once.
- Dialect-specific output should be generated by tooling.

### Fixture Compiler

Two-medium split:

- **bash** for docker lifecycle (`db:up`, `db:down`) — short shell-outs to `docker compose` plus `scripts/db/wait.sh`.
- **TypeScript** under `scripts/fixtures/` for everything else (`db:seed`, `db:reset`, `db:connections`, `db:generate`), run via `tsx`. Use `@faker-js/faker` for synthetic data (multilingual locale support out of the box: `ko`, `ja`, `zh_CN`, `en`, etc.).

Candidate structure:

```text
scripts/fixtures/
  index.ts            // CLI entry: parses `seed | reset | connections | generate` subcommands
  spec.ts             // base.yaml + profile.yaml loader + zod validation + deep merge
  generator.ts        // faker-driven row generation, FK ordering, edge integrity gating
  postgres.ts         // PG dialect: CREATE DATABASE (system DB), CREATE SCHEMA, COPY/INSERT
  mysql.ts            // MySQL dialect (placeholder until adapter activation)
  mongo.ts            // Mongo dialect: bulk insert + embed shaping
  apply.ts            // pg / mongodb client glue, profile DB ensure-exists
  connections.ts      // direct app-storage upsert (plaintext)

fixtures/
  base.yaml           // entity shape (shared)
  profiles/
    development.yaml  // primary consumer
    e2e.yaml          // built alongside dev, dormant until tauri-driver fixed
```

pnpm scripts (in `package.json`):

```json
"db:up":           "docker compose up -d postgres mongo mysql && ./scripts/db/wait.sh",
"db:down":         "docker compose down",
"db:seed":         "tsx scripts/fixtures/index.ts seed",
"db:reset":        "tsx scripts/fixtures/index.ts reset",
"db:connections":  "tsx scripts/fixtures/index.ts connections",
"db:generate":     "tsx scripts/fixtures/index.ts generate"
```

Reason:

- bash is the natural medium for docker-compose shell-outs; wrapping `docker compose up -d` in a TS `child_process.spawn` would be ceremony for no payoff.
- TypeScript is the natural medium for spec parsing (yaml + zod), faker-driven generation, and DB client work — these are what `tsx`-based scripts already do across this repo.
- Keeping fixture compilation out of the Rust crate avoids polluting `src-tauri/` with non-product binaries; reusing the existing `scripts/*.ts` lane keeps cargo build times unchanged.

### CLI Commands

```bash
pnpm db:up                                    # docker compose up + wait. profile-less.
pnpm db:down                                  # docker compose down. profile-less.
pnpm db:seed development --target all
pnpm db:reset development --target all
pnpm db:connections upsert development
pnpm db:connections clear development
pnpm db:generate e2e --target postgres
```

Command semantics:

- **`db:up`** (profile-less): bring up the docker-compose stack (postgres + mongo) and wait for healthy. Idempotent — no-op if already up. **No DB-level work** — does not create databases, does not seed.
- **`db:down`** (profile-less): `docker compose down`. Volumes are preserved; data persists. Use `docker compose down -v` directly when explicit teardown is desired.
- **`db:seed <profile>`**: ensure the profile's database exists (PG: connect to `postgres` system DB, check `pg_database`, run `CREATE DATABASE` if missing; Mongo: lazy — created on first write), then migrate (schemas/collections) and insert data. If the profile's primary entity already has rows, silently skip with `already seeded — use 'db:reset' to refill`.
- **`db:reset <profile>`**: drop + recreate the profile DB (PG: `DROP DATABASE IF EXISTS` then `CREATE DATABASE`; Mongo: `dropDatabase()` on the named DB), then migrate and insert. Always executes; explicit and destructive by name.
- **`db:connections upsert <profile>`**: write deterministic fixture connections to app storage (group `Fixtures`, ids prefixed `fixture-`, plaintext password). Independent of DB lifecycle.
- **`db:connections clear <profile>`**: remove only `fixture-*` connections from app storage.
- **`db:generate <profile>`**: print dialect output (CREATE TABLE / INSERT / `db.<col>.insertMany(...)`) to stdout for debugging. No DB connection.

Composition:

- `db:up` and `db:seed` are independent. First-time setup runs both; daily dev runs `db:up` once per session and may skip `db:seed` (silent skip).
- `db:reset` does not run `db:connections upsert`. Connection state is an app-storage concern, kept separate from DB lifecycle.
- A convenience pnpm script (e.g. `pnpm db:bootstrap development` ⇒ `db:up && db:seed development && db:connections upsert development`) is left as an optional wrapper, not a primary command.

Reason:

- Two-layer model (container vs DB) maps cleanly: `db:up`/`db:down` own the container layer; `db:seed`/`db:reset`/`db:connections` own the data layer.
- `db:seed` doing just-in-time `CREATE DATABASE` (Decision 14) keeps profile addition friction-free — no init-script drift, no manual psql step.
- Silent-skip on populated `db:seed` makes daily-dev unconditional safe.

### Populated Heuristic

`db:seed` decides "already populated" by checking the row / document count of the profile spec's first declared entity (`> 0` ⇒ populated).

Reason:

- Cheap single query, deterministic per-profile.
- Edge case (schema corrupted but data partly present) is rare; user can always run `db:reset` explicitly.

### CLI Output Format

Per-entity progress lines + start/end header. Human-readable plain text.

```
$ pnpm db:seed development --target all
db:seed development (target=all)
  postgres → table_view_development
    customers          200 rows   120ms
    products            80 rows    45ms
    orders             500 rows   210ms
    order_items      1,500 rows   480ms
  mongodb → table_view_development
    core_customers     200 docs    90ms
    catalog_products    80 docs    30ms
    sales_orders       500 docs   180ms
done — 2,560 rows + 780 docs in 1.4s
```

- One line per entity: `<entity>  <count>  <elapsed>`.
- Header on start (`db:seed development (target=all)`) and per-target subheader (`postgres → table_view_development`).
- Footer with totals + total elapsed.
- `--quiet` flag suppresses per-entity lines, keeps header + footer (useful for CI).
- TTY: optional spinner / live update; non-TTY: plain line per entity (no carriage-return tricks).
- Errors go to stderr with full stack + `exit 1`. Partial data is **not** cleaned up automatically; the user runs `db:reset <profile>` to recover. The error message includes that hint verbatim.
- Same shape applies to `db:reset` and `db:generate` (generate uses `dry-run` subheaders instead of target subheaders).

Reason:

- Per-entity lines give the user a "where did it stall?" signal during the 5-30s seed of `development`. Single-line "done" output (option a) hides which entity is slow.
- Plain text + optional `--quiet` covers human + CI reading without forking into JSON-line machinery that has no current consumer.
- Treating partial-failure cleanup as the user's responsibility (via `db:reset`) is consistent with the throwaway-data policy and keeps the seed path simple — no rollback bookkeeping.

### Schema Drift Policy

No migration tracking. The fixture is throwaway data and `db:reset <profile>` is the sole path to apply schema changes from `base.yaml`.

After pulling schema changes from git, the user runs `db:reset <profile>` manually. The CLI does not detect drift, does not version the schema, does not store hashes.

Reason:

- Fixture data is not a production migration target. Building drift-detection infra (version fields, meta tables, hash comparison) imports a mental model that does not fit the use case.
- Cosmetic YAML changes (comments, key ordering) would create false positives in any hash-based scheme; reliable detection would require a stable canonical form, which is more infrastructure than the problem warrants.
- README / handoff one-liner ("schema 바뀌면 `db:reset`") is the simplest and most honest contract. Daily dev rarely pulls schema changes; CI is unaffected because each e2e run starts with `reset`.

### PostgreSQL Mapping

PostgreSQL should use schemas:

```text
database = table_view_<profile>
logical sales.orders -> sales.orders
```

Example:

```text
core.customers
catalog.products
sales.orders
sales.order_items
ops.shipments
support.tickets
analytics.product_views
```

Reason:

- PostgreSQL users commonly use schemas.
- This tests the app's schema tree behavior directly.

### MySQL Mapping — Deferred

MySQL adapter is not active in this handoff's vertical slice. When activated, MySQL should use a single database plus table prefixes:

```text
database = table_view_<profile>
logical sales.orders -> sales_orders
```

Reason:

- Real MySQL applications usually use one database per app.
- Multiple databases just to mimic PostgreSQL schemas would be artificial.
- Prefixes provide logical grouping while preserving realistic MySQL usage.
- MySQL spec data is only generated once the adapter lands; preparing it earlier risks bitrot against an evolving spec.

### MongoDB Mapping

MongoDB should project the same logical dataset into hybrid document collections.

Collections:

```text
core_customers
catalog_products
sales_orders
catalog_reviews
support_tickets
analytics_events
inventory_movements
```

Pattern:

- `customers` and `products`: independent collections.
- `orders`: aggregate root, with items/payment/shipment embedded.
- `orders`: customer/product references plus snapshots where useful.
- `support_tickets`: messages embedded.
- `analytics_events`: event/log-style collection.

Reason:

- MongoDB should not be modeled as fully normalized RDB tables.
- Hybrid embed/reference is a common MongoDB pattern.
- Keeping the same logical dataset makes RDB/document behavior comparable.

### Multilingual Data

Each profile declares a `locale_mix` whose values sum to 1.0:

```yaml
locale_mix: { en: 0.85, ko: 0.05, ja: 0.03, zh: 0.02, edge: 0.05 }
```

**Mix is applied per-row.** Each row draws one locale once; every column on that row marked `locale_aware: true` then uses that locale. A row's columns therefore stay coherent (Korean name + Korean address) instead of mixing locales arbitrarily within a single row.

**`edge` rows** inject an edge-case value into exactly **one** string column on that row; the remaining columns use a normal locale (typically `en` or the profile's most common). Single-column injection lets sort/filter/grid regressions be attributed to a specific column.

Edge categories:

| Category | Description |
| --- | --- |
| `null` | NULL value |
| `empty` | empty string `""` |
| `emoji` | unicode emoji + mixed scripts (`"🌟⭐ 안녕 👋"`) |
| `rtl` | right-to-left scripts (Hebrew, Arabic) |
| `very_long` | 1000+ character string |
| `quotes_newlines` | escape-bait (`"line1\nline2 \"quoted\" 'apos' \t\\"`) |

### Edge Integrity Guards

Edge cases must never violate DB integrity. The compiler gates category selection per column based on column metadata:

```yaml
columns:
  id:          { type: uuid, primary: true }                          # PK: never edge
  email:       { type: email, unique: true, nullable: false }
  name:        { type: full_name, nullable: false, max_length: 200 }
  bio:         { type: text, nullable: true, max_length: 5000 }
  parent_id:   { type: ref, to: categories.id, nullable: true }
  customer_id: { type: ref, to: customers.id, nullable: false }
```

Defaults: `nullable: false`, `unique: false`, `primary: false`. PK auto-implies `nullable: false, unique: true`, and PK columns never receive edge values.

Per-category gating:

| Category | Excluded when |
| --- | --- |
| `null` | `nullable: false` |
| `empty` | `unique: true` (collision risk) or `min_length` set |
| `very_long` | `max_length` set (or truncate to limit) |
| `emoji` / `rtl` / `quotes_newlines` | always safe for string-typed columns |

Additional rules:

- **FK columns**: only `null` is allowed (when `nullable: true`); never receive string edge categories — referential integrity must hold.
- **Non-string types** (`int`, `decimal`, `timestamp`, `boolean`): only `null` edge applies (when nullable).
- **Fully-incompatible columns**: when no edge category fits, the column uses a normal value even on an edge row (no row is silently dropped).
- **Post-generation validation**: the compiler verifies every generated row against unique/nullable/max_length/FK constraints in-memory before applying. Validation failure ⇒ regenerate the row with a fresh seed offset; 3 retries then hard-fail with a spec-error message (a true failure means the spec itself is over-constrained, e.g. `unique: true` enum with too few values for the row count).

Reason:

- Per-row locale mix matches realistic data shapes; per-value mixing produces incoherent rows.
- Single-column edge injection makes regressions attributable to a specific column rather than a soup of edge effects.
- Integrity gating means fixtures load deterministically regardless of profile size; a fixture that breaks `nullable: false` would fail the apply step and stop the pipeline silently otherwise.

### Fixture Connections

Fixture connection setup is separate from product import/export.

The fixture CLI writes app storage directly:

- group name: `Fixtures`
- deterministic connection ids with `fixture-` prefix
- upsert by id
- clear only `fixture-*` connections
- preserve configured order
- store passwords as plaintext

Reason:

- Fixture connections target local docker DBs whose passwords are already public knowledge (defined in the same repo's compose file). Encrypting them adds no real security and complicates the upsert path.
- Direct fixture upsert avoids metadata fields and avoids repeated duplicate imports.
- Product-grade encrypted import/export is tracked separately and does not block fixture work.

### Product Import/Export — Out Of Scope (security-review prerequisite)

TablePlus-style import/export with password inclusion (encrypted export, preview/conflict resolution, merge/skip/rename) is needed as a product feature, but it is **not** the mechanism for fixture seed management. Fixture connections are written directly via `connections upsert` (plaintext password, deterministic id, `fixture-` prefix).

Track import/export in a **separate handoff**, and that handoff must begin with a **threat-model document** (assets / threats / current-infra audit / user-error scenarios / mitigations / residual risks) before any options-grill discussion. Reason: passwords and key material warrant rigorous prior analysis rather than UX-style trade-off recommendations. This document does not block on it.

Reason:

- Fixture automation and user-facing connection portability have different safety requirements (silent upsert vs. preview/conflict UX) and different password-handling rules (plaintext local file vs. encrypted export).
- Keeping the two tracks separate avoids coupling fixture stability to a larger product surface.
- Existing infrastructure (`aead_encrypt_with_password`, Argon2id m=64MB t=3 p=4, AES-256-GCM, BIP39 mnemonic generator) makes the *technical* path tractable, but the *policy* (which fields ship in plain JSON vs. encrypted envelope, whether plaintext export of passwords is ever allowed, how mnemonic delivery and recovery work) needs the threat model first.

### Connection Order

Keep flat groups. Do not add nested groups.

Use array order as canonical order. Do not add `sort_order` yet.

Reason:

- The current storage already preserves array order.
- Existing UI renders root connections, then groups, in stored order.
- `sort_order` would require migration and is not needed now.
- Nested groups are not planned.

### Reorder UX

Use `dnd-kit` (replacing the current HTML5 native DnD) plus a context menu fallback.

**Drag activation**: hover-revealed handle (Notion/Linear pattern). The `GripVertical` icon is hidden by default, becomes visible on row hover, and is the only `dnd-kit` listener target. Click activation on the row body is unaffected. Handle hit area is widened to ~16px to avoid mis-aim.

**Drop visualization**: no static indicator (no inserted line, no target box). Items live-shift via dnd-kit's default `arrayMove` so the visible list is itself the preview. Drop commits the current visual state.

**Drag feedback**: cursor-following `DragOverlay` only. The source row's slot collapses (dnd-kit default: source becomes invisible while the overlay carries the visual representation), so the visible row count during drag stays equal to the list size + overlay. No partial-opacity fade on the source — fade adds noise without aiding tracking.

**Connection drag patterns** (all covered by per-container `SortableContext` plus dnd-kit's hover-driven cross-container insert):

- root ↔ root reorder
- group ↔ same group reorder
- root ↔ group move + insert at hover position
- group ↔ root move + insert at hover position
- group ↔ group move + insert at hover position
- collapsed group: spring-loaded auto-expand after ~500ms hover, then insert at hover position (no append-only fallback)

**Group drag**: groups reorder only (no nesting; flat groups per `Connection Order` decision).

**Auto-scroll**: when the cursor enters the top or bottom 40px of the sidebar viewport during a drag, scroll in that direction. Speed scales linearly with proximity to the edge (8 → 24 px/frame). Stops at scroll limits. Implemented via `useEffect` + `requestAnimationFrame` listening to dnd-kit's active drag state.

**Keyboard a11y (full support)**:

- Drag handle is `tabindex=0`. Tab navigates to it; visible focus ring matches the project's standard.
- Space / Enter on the focused handle activates dnd-kit's `KeyboardSensor`. Arrow Up/Down moves, Arrow Left/Right cross-container moves (dnd-kit default `sortableKeyboardCoordinates`).
- Space / Enter again drops; Esc cancels and restores original position.
- dnd-kit's screen-reader announcements are customised to project copy ("Picked up `<connection name>`", "Dropped into `<group>` at position N", etc.).

The context menu fallback (Decision 5) remains available alongside keyboard drag, so users can choose either model.

**Context menu fallback**:

Connection (right-click on connection row):

- Move Up / Move Down / Move to Top / Move to Bottom — within the same container only (root or current group). Use "Move to Group" to leave the container.
- ─────
- Move to Group → submenu:
  - Current group: shown with checkmark, disabled (so the user can confirm where they are without it being clickable).
  - Other groups: enabled, click to move.
  - ─────
  - `(Root, no group)`: enabled when connection is currently in a group.
  - ─────
  - `+ New group…`: opens the existing new-group dialog and moves the connection into the new group on confirm.
- Disabled-but-visible: when the connection is already at the top of its container, "Move Up" / "Move to Top" render disabled (greyed) instead of being hidden, so muscle memory does not break.

Group (right-click on group header):

- Move Up / Move Down / Move to Top / Move to Bottom — within the group ordering only.
- ─────
- Rename / Change Color (unchanged from current behaviour).
- Collapse / Expand (mirrors the chevron toggle).
- ─────
- Delete Group (unchanged).

Reason:

- The hover-revealed handle removes the click/drag conflict that the current always-visible handle introduces, and removes visual clutter from the sidebar.
- Live-shift drop visualization avoids a separate indicator language; the list itself is the source of truth.
- Spring-loaded collapsed-group expansion is the Finder/Explorer pattern; users can position precisely instead of needing a second drop to fix append order.
- Context fallback preserves precision and accessibility (keyboard-only) without adding more drag affordances.

### Reorder Backend

One command saves the entire list shape — group order, connection order, and group assignment in a single atomic write:

```rust
#[derive(Deserialize)]
pub struct ConnectionPlacement {
    pub id: String,
    pub group_id: Option<String>,
}

#[tauri::command]
pub fn reorder_connection_list(
    group_ids: Vec<String>,
    connections: Vec<ConnectionPlacement>,
) -> Result<(), AppError>
```

Behavior:

- `group_ids` is the **complete** final order of all groups; missing or unknown ids ⇒ `ValidationError`.
- `connections` is the **complete** final order of all connections; missing or unknown ids ⇒ `ValidationError`. Each placement's `group_id` is the connection's new group (or `None` for root).
- Acquires `STORAGE_LOCK`, validates, rewrites the in-memory `Vec`s, then atomic-writes `connections.json`.
- Single-window app + atomic-rename file write means last-write-wins is acceptable; the frontend can recover from any race by reloading via `list_connections()`.

Reason:

- Implicit "append missing items at the end" was rejected — it hides bugs (frontend dropped an id) instead of surfacing them. Strict length match is louder and easier to debug.
- One command covers group reorder, connection reorder, and cross-container moves; the optimistic frontend state then mirrors a single IPC round-trip.
- Backend remains the SoT (per ADR 0021 / AC-142 multi-window sync); frontend proposes the new order, backend validates and persists.

### Spec File Layout

Split the spec into a base entity definition and per-profile delta files:

```text
fixtures/
  base.yaml              # entity shape: targets, columns, mongo embed, relationships
  profiles/
    e2e.yaml             # seed + locale_mix + per-entity row counts
    development.yaml
```

`base.yaml` is the source of truth for *what entities exist and what their columns look like*. Each `profiles/<profile>.yaml` overrides only *size and distribution* — row counts per entity, multilingual/edge mix, deterministic seed. The compiler deep-merges base + profile and validates (zod schema). A profile that omits a row count for an entity defined in base is a validation error.

Reason:

- Profile-vs-entity is a clean two-axis decomposition. Profiles vary size and distribution; the entity shape itself does not change between profiles. Encoding both in a single file forces duplication every time an entity is added.
- Adding a new entity is a base-only edit plus one row-count line per profile, instead of touching every profile file.
- Reviewing changes is profile-scoped or shape-scoped, never both at once.

### Column Type Vocabulary

Use an abstract type vocabulary instead of binding spec values directly to `@faker-js/faker` API names.

```yaml
columns:
  email: { type: email }
  name:  { type: full_name, locale_aware: true }
```

The compiler maps each abstract type (`uuid`, `email`, `full_name`, `product_name`, `sku`, `phone`, `address`, `decimal`, `int`, `timestamp`, `enum`, `array_of`, `ref`, `text`, `boolean`, `json`) to its faker call (or to a non-faker source, e.g. regex / fixed list / NULL injection) in one place.

Reason:

- Spec stays stable across faker major versions; a faker 8→9 breaking change is absorbed by the mapping layer rather than 100+ spec lines.
- The vocabulary doubles as a contract: abstract types carry semantic meaning (`email` always validates, `sku` always matches the SKU regex) where raw faker names would not.

### Foreign Key Distribution

References are sampled with a uniform random distribution.

```yaml
order: { customer_id: { type: ref, to: customers.id } }
```

Reason:

- The fixture's primary purpose is to exercise the DB client (encoding, sort/filter, paging, joins) — realistic data-distribution shape is secondary.
- Distribution options (zipf, normal, hot/cold) can be added later as `distribution: …` without breaking existing specs.

### MongoDB Embed Cardinality

Embedded relationships must declare cardinality explicitly.

```yaml
mongo:
  embed:
    items:   { from: order_items, kind: many }
    payment: { from: payments,    kind: one  }
```

The compiler emits an array (`[ … ]`) for `kind: many` and a single object for `kind: one`. Auto-inference from PK uniqueness was rejected.

Reason:

- A one-line declaration is cheaper than a generator that has to guess and occasionally guess wrong.
- The cardinality choice belongs to the fixture author, not the compiler.

### Host Port Configuration

Fixture connections read host/port from the **same environment variables** that drive `docker-compose.yml`:

| Variable | Purpose | Default |
| --- | --- | --- |
| `PGHOST` | postgres host (CLI usage) | `localhost` |
| `PGPORT` | postgres host port | `15432` |
| `PGUSER` | postgres user | `testuser` |
| `PGPASSWORD` | postgres password | `testpass` |
| `MONGO_HOST` | mongo host | `localhost` |
| `MONGO_PORT` | mongo host port | `37017` |
| `MONGO_USER` | mongo root user | `testuser` |
| `MONGO_PASSWORD` | mongo root password | `testpass` |

`db:connections upsert <profile>` reads these (with the documented defaults) and writes `host:port` into each fixture connection. When the user overrides `PGPORT=15433`, the next `db:connections upsert` writes `15433` into the upserted fixture connections — the docker-compose binding and the app-storage entry stay aligned via the same SoT.

The e2e service-container path (`PGHOST=postgres`, `PGPORT=5432`) is unaffected: the fixture CLI runs on the host, not inside the docker network, so it never reads the in-network defaults.

Reason:

- `docker-compose.yml` already treats `PGPORT` / `MONGO_PORT` as the fact-of-the-cluster. Reusing the same variable is single SoT instead of two parallel knobs.
- Daily dev uses the defaults; only port-collision cases require the override, and that one `export PGPORT=…` covers both compose and fixture connections.

## Implementation Order

### 1. Fixture Compiler — Development + E2E (vertical slice)

Build the smallest end-to-end path covering both profiles in one slice:

- `base.yaml` + `profiles/development.yaml` + `profiles/e2e.yaml`.
- TypeScript CLI (`scripts/fixtures/`) for PostgreSQL and MongoDB.
- bash wrappers / pnpm scripts for `db:up` / `db:down`.
- Fixture connection direct upsert (plaintext) for both profiles.
- Add mongo volume to `docker-compose.yml`.

Reason:

- The `development` profile is the immediate consumer (daily dev workflow); the e2e WebDriver flow is currently blocked by tauri-driver cold-start + OOM, so e2e fixture data has no live consumer right now. We still build `profiles/e2e.yaml` because the engine cost is the same (it is one extra small file once the compiler exists), and once the driver issue is resolved the fixture is already in place.
- `table_view_test` (cargo integration tests) is **untouched** — those tests still use their existing seed flow. Migration to the new compiler is a separate follow-up.
- `e2e/fixtures/seed.sql` is **untouched** for the same reason.

### 2. Connection Ordering
