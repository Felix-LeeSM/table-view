### Assumptions

- **Sensitive key choice — `loading`/`error` rather than `password`**. The store does not carry a top-level `password` field; passwords live nested inside `connections[i].password`, but the backend redacts them via `has_password: boolean` before serialization (see `ConnectionConfig` type), so the `connections` array is already password-free on the wire. The contract's "sensitive key" assertion was therefore exercised against the next-best transient/window-local fields the store actually has: `loading` (in-flight `loadConnections()` flag) and `error` (last-error string). The c-out, c-in, c-mix cases pin exactly these two as non-broadcast; the c-mix case asserts the wire payload contains exactly the four allowlisted keys and nothing else. If a future store carries a top-level secret, this same pattern locks it out.
- **Origin id fallback `"test"`**. `getCurrentWindowLabel()` returns `null` outside a Tauri runtime (vitest jsdom). The store falls back to `"test"` so module load doesn't crash. Tests `vi.mock("@lib/window-label")` to override per-scenario (`"launcher"` / `"workspace"`). In production both windows always return their real label.
- **Bridge attach is module-load `void`-ed**. The Promise returned by `attachZustandIpcBridge` is not awaited (a synchronous module-load site can't await). Outbound emits work synchronously via the store-subscribe; inbound apply may race the `listen()` resolution on the very first event in production. Both windows go through the same race window, so net behavior converges on the next mutation. For tests, `vi.hoisted()` materializes the bus map before module load and the bridge's `listen` resolves synchronously through the mock — so no race in tests either.
- **`vi.hoisted()` pattern (vs the colocated `vi.mock` factory in `zustand-ipc-bridge.test.ts`)**. The bridge test attaches the bridge MANUALLY inside `it(...)`, so by then the closure-captured `bus` Map exists. This sprint's test file imports `connectionStore`, which calls `attachZustandIpcBridge` AT MODULE LOAD — before the file's other top-level `const bus = new Map()` initialization runs. `vi.hoisted()` is the contract-prescribed remedy. Documented in the test file's bus-section comment.
- **`emit` mock added to `connectionStore.test.ts`**. Pre-Sprint-152, the existing mock only declared `listen`. The bridge's outbound path calls `emit` at module load and would throw "No emit export defined" without this. The added mock is a no-op `vi.fn(() => Promise.resolve())` — these tests don't assert the cross-window contract (that's the dedicated test file's job); they only need the bridge to import cleanly.
- **No subscription cleanup**. The bridge's dispose function is not retained — bridge lives for the renderer process lifetime. Tests reset state via `useConnectionStore.setState(...)` in `beforeEach` (existing pattern). The bus from the cross-window test file is also persistent across the file's tests (we deliberately do NOT call `busReset` in `beforeEach` because that would unregister the module-load-attached listener); state leakage across tests is prevented by `resetStore()` and `mockedEmit.mockClear()`.

### Residual Risk

- **`vi.hoisted` introduces test-file/bridge coupling**. The cross-window-sync test file's bus must be hoisted because the bridge attach runs at module load. If a future sprint moves the bridge attach to be lazy (e.g. invoked from `App.tsx` after window-label resolution rather than at import time), the `vi.hoisted` pattern would still work but the `vi.mock` factory could be simplified back to the colocated style. Not blocking.
- **Bridge attach failure is silently swallowed via `.catch(() => {})`**. If `listen` rejects (e.g. Tauri runtime missing), the store still functions window-local. We do not surface this to the user — same trade-off the bridge module made for emit failures (Sprint 151 assumption #4). Sprint 154's e2e gate is the first time real Tauri `listen` is exercised; if it ever fails, the launcher and workspace silently desynchronize. A future tightening could log to `console.warn` in non-test environments.
- **Inbound apply triggers a full subscriber re-render in real Zustand-React stores** (carried over from Sprint 151 residual #1). The bridge calls `store.setState(filtered)`; React subscribers re-render exactly as if local code had set the same fields. Sprint 153 should audit per-store equality fns when wiring `tabStore`/`mruStore` (higher-frequency stores).
- **Self-origin loop guard relies on the window label being unique across windows**. `tauri.conf.json` declares `launcher` and `workspace` labels. If a future sprint adds a third window with a non-unique label by accident, that window's emits would echo back without being filtered. Not a current bug; worth flagging when Sprint 153/154 land more windows.
- **Sprint 150 boot-duplication (Sprint 151 residual #5) is NOT addressed by this sprint** — the contract scope is "wire connectionStore", not "retire the workspace's duplicate fan-out". That clean-up belongs to Sprint 153/154. No regression: the bridge tolerates redundant `loadConnections` calls (they trigger a no-op emit if the result is shallow-equal to lastBroadcast).
- **TDD red-state log was captured with the pre-`vi.hoisted` version of the test file**, so 6/9 tests passed (the no-emit / no-state-change assertions are trivially true with no bridge). The 3 failing tests are the wiring-specific ones (a, b, c-mix), which is sufficient TDD evidence — the wiring would be needed regardless of the bus-hoisting strategy. Recapturing the log against the final test file would show the same 3 failures (the bus-hoisting is an implementation detail of the TEST setup, not the production wiring under test). Header in the log file documents this.
