# Sprint Contract: sprint-393a

## Summary

- Goal: **SELECT widening — pass 1 (read-only relational core)** — extend the
  sprint-385 / sprint-391 / sprint-392 grammar with the foundational SELECT
  features that almost every read query uses: multi-table FROM, JOIN family,
  WHERE expression widening (column-op-column / BETWEEN / LIKE-family),
  GROUP BY, HAVING, ORDER BY, and LIMIT/OFFSET. sqlSafety classification of
  these read-only statements stays `select` / `severity: info` — the value
  delivered by this sprint is *grammar coverage*, not new safety rules.
- Audience: sprint-393b (CTE / window / subquery / UNION / CASE / IN-list)
  builds on top, then 394 / 395. The WHERE expression widening introduced here
  is the same expression grammar reused by HAVING and (later) by the DML WHERE
  in 393b regression tests.
- Owner: Generator (sprint-393a).
- Verification Profile: `frontend` (pnpm vitest + pnpm tsc + pnpm lint) +
  `backend` (`cargo test`, `cargo clippy --all-targets --all-features
  -D warnings`, `cargo build --target wasm32-unknown-unknown --release
  --features wasm`).

## Background

- sprint-385 introduced a deliberately narrow SELECT slice
  (`SELECT col-list FROM single-table [WHERE column-op-literal]`).
- sprint-392 introduced a *minimal* WHERE expression (column-op-literal +
  AND / OR / NOT / IS NULL / IS NOT NULL) shared by UPDATE and DELETE.
  Anything outside that slice was returned as `Error(UnsupportedExpression)`.
- Real-world SELECT queries — even the simplest dashboards generated by the
  app — need multiple tables, joins, BETWEEN, LIKE, GROUP BY, ORDER BY, and
  LIMIT. The sprint-385 grammar rejects all of these, so the AST path of
  `sqlSafety` falls back to the regex fast-path for every non-trivial SELECT.
- This sprint closes that gap for *read-only flat queries*. Set operations,
  CTE wrapping, window functions, subqueries, and CASE are intentionally
  deferred to sprint-393b so that this sprint can be merged independently
  without an oversized review surface.
- sqlSafety severity for the surface introduced here remains `info`. No new
  destructive classification appears — the AST simply gains the ability to
  recognize more shapes as a valid SELECT and to keep returning the existing
  `select` severity instead of an `Error(UnsupportedExpression)` fallback.

## In Scope

### Lexer additions

The lexer gains the following new case-insensitive keyword tokens. They are
recognized identically to existing keywords (`Select`, `From`, `Where`) — the
same uppercase / lowercase / mixed-case handling already used by sprint-385.

- Join family: `JOIN`, `INNER`, `LEFT`, `RIGHT`, `FULL`, `OUTER`, `CROSS`,
  `ON`, `USING`.
- Predicate keywords: `BETWEEN`, `LIKE`, `ILIKE`, `NOT` (already lexed by
  sprint-392 — reused, not duplicated), `AS` (used by alias).
- Grouping / ordering / paging keywords: `GROUP`, `BY`, `HAVING`, `ORDER`,
  `ASC`, `DESC`, `NULLS`, `FIRST`, `LAST`, `LIMIT`, `OFFSET`.
- Punctuation: `.` (period — used by schema qualifier and qualified column
  reference). The lexer must distinguish `.` between identifiers (qualifier)
  from any other context; for this sprint it appears only between two
  identifier tokens.

All other characters / keywords behave exactly as in sprint-392. A statement
that uses a keyword not yet introduced still results in a parse error
(`Error(UnsupportedExpression)` or `SyntaxError`, depending on position) so
that 393b can extend the grammar without breaking 393a's invariants.

### AST additions (natural-language spec)

The `ParseResult` discriminated union does NOT gain a new top-level variant
in this sprint — `SELECT` statements still serialize under the existing
`kind: "select"` tag introduced by sprint-385. The widening is expressed by
adding optional fields and richer nested shapes inside the existing
`SelectStatement` shape.

- `SelectStatement` carries an *ordered list of FROM items* in place of the
  sprint-385 single table reference. Each FROM item has three properties:
  a fully-resolved table reference (schema-qualifier optional, base name
  required), an optional alias (string, only present when `AS alias` or a
  bare identifier alias was supplied), and a join descriptor describing how
  this item attaches to the previous item.
- The join descriptor variants identify themselves by a kebab-case `kind`
  string: `comma` (the first item and any later comma-separated item),
  `inner-join`, `left-join`, `right-join`, `full-join`, and `cross-join`.
  Every join variant other than `comma` and `cross-join` carries a join
  predicate. The predicate variants identify themselves by a kebab-case
  `kind` string of `on` (carrying a WHERE-style expression — see below) or
  `using` (carrying an ordered list of column identifiers).
- The `SelectStatement` gains four optional clause slots, each absent
  (serialized as `null` or empty list — see Invariants) when not supplied by
  the input: a *GROUP BY* slot holding an ordered list of column references
  (qualified column references allowed — `table.column`); a *HAVING* slot
  holding the same expression shape used by WHERE; an *ORDER BY* slot
  holding an ordered list of ordering items; and a *LIMIT* slot holding a
  count and an optional offset.
- An ordering item carries a column reference (qualified column allowed),
  an explicit direction tag (`asc` or `desc`; `asc` when the direction was
  omitted in source), and a nulls-placement tag (`first`, `last`, or
  `unspecified`). The tag is `unspecified` exactly when the input did not
  contain a `NULLS FIRST` or `NULLS LAST` token. The serializer must not
  silently default `unspecified` to `first` or `last` — downstream tooling
  reads it directly.
- The `LIMIT` slot holds a count value of literal-or-placeholder shape (the
  same `InsertValue` shape introduced by sprint-392) and an offset value of
  the same shape that is absent (null) when `OFFSET` was not supplied.

The expression shape used by WHERE is widened — see "Expression widening"
below. The same widened shape is reused by HAVING and by any JOIN `ON`
predicate. There is no separate AST node for "JoinExpr" or "HavingExpr"; all
three sites embed the WHERE expression directly.

#### Expression widening

The sprint-392 WHERE expression supported only column-op-literal comparisons
plus AND / OR / NOT / IS NULL / IS NOT NULL. This sprint extends that
expression with three new primary forms:

- **Column-op-column comparison.** The right-hand side of an existing
  comparison may now be a qualified or unqualified column reference instead
  of an `InsertValue`. The AST gains a new comparison primary that records
  `left` (column reference), `op` (the same six-op compare op set), and
  `right` (column reference). A literal vs. column comparison continues to
  use the sprint-392 `comparison` primary.
- **BETWEEN.** A predicate of the form `column BETWEEN value1 AND value2`
  parses to a primary identified by `kind: "between"` carrying a column
  reference, a low bound, and a high bound; both bounds use the same
  literal-or-placeholder shape as the existing comparison. The negated form
  `column NOT BETWEEN value1 AND value2` parses to the same shape wrapped in
  the existing `not` primary, not to a separate `not-between` variant —
  callers can unambiguously test `kind` without enumerating "negative twins".
- **LIKE / ILIKE.** Predicates of the form `column LIKE 'pattern'` and
  `column ILIKE 'pattern'` parse to a primary identified by `kind: "like"`
  carrying a column reference, a case-sensitivity tag (`sensitive` for
  `LIKE`, `insensitive` for `ILIKE`), and a pattern value of the same
  literal-or-placeholder shape. `column NOT LIKE 'p'` and `column NOT ILIKE
  'p'` parse to the `like` primary wrapped in `not`.

A column reference primary in either side of a comparison serializes with
two properties — `table` (string or null when unqualified) and `column`
(string). Identifier resolution / scoping is *not* the parser's job; the
parser only records what the input wrote.

The widened expression *adds* to the sprint-392 grammar without changing
existing variants. Statements that parsed under sprint-392 continue to
parse identically here.

### Parser additions

`parse_select` (the existing sprint-385 entry) is widened, NOT replaced. The
following sub-parsers are added or extended:

- **FROM list parser** — parses one or more FROM items separated by commas
  or by join keywords. Items separated by commas are equivalent to a CROSS
  JOIN per SQL semantics, but this sprint preserves the source style: a
  `,`-separated item serializes its join descriptor as `kind: "comma"`, and
  a syntactic `CROSS JOIN` serializes as `kind: "cross-join"`. The two AST
  shapes are *not* normalized to one — downstream tooling must accept both.
- **Schema-qualified table reference** — accepts `identifier.identifier` as
  a table reference. The first identifier becomes the optional `schema` slot
  on the FROM item; the second becomes the `table` slot. A bare identifier
  leaves `schema` null.
- **Alias parser** — after a table reference (or after a join, before its
  predicate), accepts an optional `AS identifier` or bare `identifier` as
  the alias of the immediately preceding FROM item.
- **Join clause parser** — recognizes `[INNER] JOIN`, `LEFT [OUTER] JOIN`,
  `RIGHT [OUTER] JOIN`, `FULL [OUTER] JOIN`, and `CROSS JOIN`. Every variant
  except `CROSS JOIN` then expects either `ON <expression>` or `USING
  (<column-list>)`. `CROSS JOIN` has no predicate.
- **Expression widening** — the WHERE primary parser gains the three new
  primaries (column-column comparison, BETWEEN, LIKE/ILIKE) listed above.
  Precedence: `OR < AND < NOT < primary`; BETWEEN and LIKE bind at the
  primary level (same level as the existing comparison). NOT BETWEEN /
  NOT LIKE are recognized as `NOT <primary>` rather than as discrete forms.
- **GROUP BY parser** — after WHERE (or after FROM if WHERE is absent),
  optionally accepts `GROUP BY <column-ref-list>`. A column reference list
  is one or more qualified-or-unqualified columns separated by commas.
- **HAVING parser** — only valid when GROUP BY is present. Accepts
  `HAVING <expression>` using the widened expression grammar. A HAVING
  without GROUP BY parses to `SyntaxError`.
- **ORDER BY parser** — accepts `ORDER BY <ordering-item-list>`. Each
  ordering item is `<column-ref> [ASC|DESC] [NULLS FIRST|NULLS LAST]`.
- **LIMIT parser** — accepts `LIMIT <value> [OFFSET <value>]`. Value must
  be an `InsertValue` (integer literal or placeholder). A `LIMIT` without
  a value, or `OFFSET` without `LIMIT`, parses to `SyntaxError`.

The clause order enforced by the parser is the standard SQL order:
`SELECT … FROM … WHERE … GROUP BY … HAVING … ORDER BY … LIMIT …`. Inputs
that supply clauses out of order (e.g. `LIMIT … ORDER BY …`) parse to
`SyntaxError` even when the unordered form is accepted by some dialects.

### TS facade updates

`src/lib/sql/sqlAst.ts` extends the existing `SqlSelectStatement` shape
with the new optional fields (FROM list, GROUP BY, HAVING, ORDER BY, LIMIT)
and exposes the new sub-shapes (FROM item with alias / join descriptor,
column reference, ordering item, LIMIT slot, expression-widening primaries).
The `kind` discriminator at the top level stays `"select"` — no new
top-level union member is added. The runtime guard `isSqlParseResult`
accepts the wider shape unchanged.

`parseSqlPreloaded` continues to work for SELECT — the safety classifier
already treats the result as `select` regardless of the wider shape.

### sqlSafety integration

`src/lib/sql/sqlSafety.ts` requires no behavior change. The function continues
to classify any successfully-parsed SELECT (whether the sprint-385 narrow
slice or the sprint-393a wide slice) as `kind: "select"` with
`severity: "info"` and an empty `reasons` array. The widening is invisible
to the safety classifier surface — it only means the regex fallback path is
exercised less often, because more SELECT shapes now resolve through the
AST branch.

There is no decision (D1/D2/etc.) about severity escalation in this sprint —
read-only SELECT remains `info` even when it joins many tables, groups, or
limits. Any escalation rule (e.g. "warn on missing LIMIT on huge tables")
is a separate sprint and not in scope.

## Out of Scope

- **CTE / WITH** — sprint-393b.
- **Window functions** (`OVER (PARTITION BY ... ORDER BY ...)`) —
  sprint-393b.
- **Subqueries** in WHERE / FROM / SELECT — sprint-393b.
- **Set operations** (`UNION`, `UNION ALL`, `INTERSECT`, `EXCEPT`) —
  sprint-393b.
- **`CASE WHEN ... THEN ... ELSE ... END`** — sprint-393b.
- **IN-list** (`WHERE col IN (1, 2, 3)`) — sprint-393b. Inputs that contain
  IN-list continue to surface as `Error(UnsupportedExpression)`.
- **Aggregate function calls** (`COUNT(*)`, `SUM(x)`) in SELECT list or
  HAVING. Without aggregates, GROUP BY / HAVING are syntactically valid but
  semantically narrow — the parser accepts them. Aggregates land in 393b.
- **Arithmetic expressions** in WHERE / SELECT (`a + b`, `a * 2`). Stays as
  `Error(UnsupportedExpression)`. 393b expands this together with CASE.
- **DDL additive** — sprint-394.
- **GRANT / REVOKE / EXPLAIN / SHOW / SET / COPY / COMMENT** — sprint-395.
- **Dialect-specific tokens** — MySQL backtick identifiers, MSSQL bracket
  identifiers, PostgreSQL `"double-quoted"` identifiers — sprint-396+.
- **NATURAL JOIN** and **LATERAL JOIN** — deferred to a later sprint with
  the rest of the join variants.
- **`LIMIT … BY` (ClickHouse / others) and `LIMIT n, m` (MySQL legacy)** —
  this sprint only accepts the ANSI `LIMIT <n> [OFFSET <m>]` shape.

## Invariants

- The TOP-LEVEL `kind` discriminator for a successful SELECT parse is
  `"select"`, unchanged from sprint-385. Downstream consumers that only
  branched on top-level `kind` need no code change.
- All sprint-385 / sprint-391 / sprint-392 tests pass without modification —
  the widening adds to the grammar without changing existing AST shapes for
  inputs that those sprints' tests already cover.
- `analyzeStatement`'s return `StatementAnalysis` shape (`kind`, `severity`,
  `reasons`) is unchanged. Existing sqlSafety tests remain green.
- The Rust crate continues to have no Tauri / tokio / IO dependency. WASM
  build (`cargo build --target wasm32-unknown-unknown --release --features
  wasm`) continues to succeed.
- No `unwrap()` / `expect()` on user-input paths. TS facade uses `unknown`
  + runtime guards; no `any`.
- WASM bundle size: ≤ sprint-392 gzipped size × 1.4. The new grammar is
  larger than the DML triad, so a higher headroom multiplier is set, but
  the budget remains explicit so reviewers can flag a runaway.
- `Error(UnsupportedExpression)` continues to be the catch-all for forms
  that this sprint deliberately defers (IN-list, subquery, aggregate,
  arithmetic). The parser must not silently accept such forms by reusing a
  near-match production.

## Acceptance Criteria

### A — FROM clause widening (AC-393a-A)

- `AC-393a-A01` Statement `SELECT a FROM x, y` parses as `kind=select` with
  a FROM list of length 2 whose second item has join descriptor
  `kind=comma`.
- `AC-393a-A02` Statement `SELECT a FROM public.users` parses with a single
  FROM item whose schema slot is `"public"` and table slot is `"users"`.
- `AC-393a-A03` Statement `SELECT a FROM users AS u` parses with a single
  FROM item whose alias slot is `"u"`.
- `AC-393a-A04` Statement `SELECT a FROM users u` (bare alias, no `AS`)
  parses with the same alias slot `"u"` as the explicit-`AS` form.
- `AC-393a-A05` Statement `SELECT a FROM public.users AS u, public.orders o`
  parses with two items, both schema-qualified, both aliased, second item's
  join descriptor `kind=comma`.
- `AC-393a-A06` Statement `SELECT a FROM` (no table) parses as
  `Error(SyntaxError)`.
- `AC-393a-A07` Statement `SELECT a FROM public.` (qualifier with no
  table) parses as `Error(SyntaxError)`.
- `AC-393a-A08` Statement `SELECT a FROM public.users.extra` (three-dot
  qualifier) parses as `Error(SyntaxError)`.

### B — JOIN family (AC-393a-B)

- `AC-393a-B01` Statement `SELECT a FROM x JOIN y ON x.id = y.x_id` parses
  with second FROM item's join descriptor `kind=inner-join` and predicate
  `kind=on` carrying a column-column comparison.
- `AC-393a-B02` Statement `SELECT a FROM x INNER JOIN y ON x.id = y.x_id`
  is parsed identically to `AC-393a-B01` (the `INNER` keyword is optional —
  it does not appear as a separate AST flag).
- `AC-393a-B03` Statement `SELECT a FROM x LEFT JOIN y ON x.id = y.x_id`
  parses with join descriptor `kind=left-join`.
- `AC-393a-B04` Statement `SELECT a FROM x LEFT OUTER JOIN y ON x.id =
  y.x_id` parses identically to `AC-393a-B03` (the `OUTER` keyword is
  optional).
- `AC-393a-B05` Statement `SELECT a FROM x RIGHT JOIN y ON x.id = y.x_id`
  parses with join descriptor `kind=right-join`.
- `AC-393a-B06` Statement `SELECT a FROM x FULL JOIN y ON x.id = y.x_id`
  parses with join descriptor `kind=full-join`.
- `AC-393a-B07` Statement `SELECT a FROM x CROSS JOIN y` parses with join
  descriptor `kind=cross-join` and no predicate field set.
- `AC-393a-B08` Statement `SELECT a FROM x JOIN y USING (id)` parses with
  predicate `kind=using` and an ordered column list of length 1.
- `AC-393a-B09` Statement `SELECT a FROM x JOIN y USING (id, tenant_id)`
  parses with predicate `kind=using` and a column list of length 2.
- `AC-393a-B10` Statement `SELECT a FROM x JOIN y` (no predicate) parses
  as `Error(SyntaxError)`.
- `AC-393a-B11` Statement `SELECT a FROM x CROSS JOIN y ON x.id = y.x_id`
  parses as `Error(SyntaxError)` (CROSS JOIN cannot have a predicate).
- `AC-393a-B12` Statement `SELECT a FROM x JOIN y ON 1 = 1` parses with
  predicate `kind=on` carrying a column-vs-literal comparison (still allowed
  — the join predicate inherits the same widened WHERE grammar).
- `AC-393a-B13` Three-table chain `SELECT a FROM x JOIN y ON x.id = y.x_id
  LEFT JOIN z ON y.id = z.y_id` parses with a three-item FROM list; the
  second item is `inner-join`, the third is `left-join`.

### C — WHERE expression widening (AC-393a-C)

- `AC-393a-C01` Statement `SELECT a FROM x WHERE x.a = y.b` parses with
  WHERE primary `kind=column-comparison` recording two qualified column
  references.
- `AC-393a-C02` Statement `SELECT a FROM x WHERE x.a > 10` continues to
  parse with WHERE primary `kind=comparison` recording a column reference
  on the left and an integer literal on the right (unchanged from sprint-392
  behavior).
- `AC-393a-C03` Statement `SELECT a FROM x WHERE x.age BETWEEN 18 AND 65`
  parses with WHERE primary `kind=between` recording the column and two
  integer bounds.
- `AC-393a-C04` Statement `SELECT a FROM x WHERE x.age NOT BETWEEN 18 AND
  65` parses with a `not` primary wrapping a `between` primary.
- `AC-393a-C05` Statement `SELECT a FROM x WHERE x.name LIKE 'fe%'` parses
  with WHERE primary `kind=like`, case-sensitivity tag `sensitive`, and
  string-literal pattern `"fe%"`.
- `AC-393a-C06` Statement `SELECT a FROM x WHERE x.name ILIKE 'FE%'`
  parses with `kind=like` and case-sensitivity tag `insensitive`.
- `AC-393a-C07` Statement `SELECT a FROM x WHERE x.name NOT LIKE 'a%'`
  parses with a `not` primary wrapping a `like` primary.
- `AC-393a-C08` Statement `SELECT a FROM x WHERE x.a = y.b AND x.c <> y.d`
  parses with an `and` primary whose left and right are both
  `column-comparison` primaries.
- `AC-393a-C09` Statement `SELECT a FROM x WHERE id IN (1, 2, 3)` parses
  as `Error(UnsupportedExpression)` — IN-list is sprint-393b.
- `AC-393a-C10` Statement `SELECT a FROM x WHERE x.age BETWEEN 18` (missing
  high bound) parses as `Error(SyntaxError)`.

### D — GROUP BY / HAVING (AC-393a-D)

- `AC-393a-D01` Statement `SELECT a FROM x GROUP BY a` parses with a
  GROUP BY list of length 1 containing one unqualified column reference.
- `AC-393a-D02` Statement `SELECT a FROM x GROUP BY x.a, x.b` parses with
  a GROUP BY list of length 2 containing two qualified column references.
- `AC-393a-D03` Statement `SELECT a FROM x GROUP BY a HAVING x.a > 10`
  parses with both GROUP BY and HAVING populated; HAVING carries the same
  primary as the equivalent WHERE.
- `AC-393a-D04` Statement `SELECT a FROM x HAVING x.a > 10` (HAVING without
  GROUP BY) parses as `Error(SyntaxError)`.
- `AC-393a-D05` Statement `SELECT a FROM x GROUP BY` (no columns) parses
  as `Error(SyntaxError)`.

### E — ORDER BY / LIMIT (AC-393a-E)

- `AC-393a-E01` Statement `SELECT a FROM x ORDER BY a` parses with an
  ORDER BY list of length 1 whose single item has direction `asc` and
  nulls-placement `unspecified`.
- `AC-393a-E02` Statement `SELECT a FROM x ORDER BY a DESC` parses with the
  item's direction set to `desc`.
- `AC-393a-E03` Statement `SELECT a FROM x ORDER BY a ASC NULLS FIRST`
  parses with direction `asc` and nulls-placement `first`.
- `AC-393a-E04` Statement `SELECT a FROM x ORDER BY a DESC NULLS LAST`
  parses with direction `desc` and nulls-placement `last`.
- `AC-393a-E05` Statement `SELECT a FROM x ORDER BY x.a, x.b DESC` parses
  with two items, the first defaulting to `asc`-`unspecified`, the second
  `desc`-`unspecified`.
- `AC-393a-E06` Statement `SELECT a FROM x LIMIT 10` parses with LIMIT
  count value `kind=literal` / integer value `10` and offset absent.
- `AC-393a-E07` Statement `SELECT a FROM x LIMIT 10 OFFSET 20` parses
  with offset value present (`kind=literal`, integer `20`).
- `AC-393a-E08` Statement `SELECT a FROM x LIMIT $1 OFFSET $2` parses
  with both slots holding `kind=placeholder` values.
- `AC-393a-E09` Statement `SELECT a FROM x LIMIT 10, 20` (MySQL legacy
  comma form) parses as `Error(SyntaxError)`.
- `AC-393a-E10` Statement `SELECT a FROM x OFFSET 20` (OFFSET without
  LIMIT) parses as `Error(SyntaxError)`.

### F — Clause ordering (AC-393a-F)

- `AC-393a-F01` Statement `SELECT a FROM x WHERE x.a > 1 GROUP BY a
  HAVING a > 0 ORDER BY a LIMIT 5 OFFSET 1` parses with every clause slot
  populated in the documented order.
- `AC-393a-F02` Statement `SELECT a FROM x ORDER BY a WHERE x.a > 1`
  (WHERE after ORDER BY) parses as `Error(SyntaxError)`.
- `AC-393a-F03` Statement `SELECT a FROM x LIMIT 10 ORDER BY a` (LIMIT
  before ORDER BY) parses as `Error(SyntaxError)`.

### S — Serialization (AC-393a-S)

- `AC-393a-S01` A successful parse of a SELECT containing every new clause
  serializes to a JSON object whose top-level `kind` is `"select"` and
  whose new slots use the documented kebab-case `kind` discriminators
  (`comma`, `inner-join`, `left-join`, `right-join`, `full-join`,
  `cross-join`, `on`, `using`, `between`, `like`, `column-comparison`).
- `AC-393a-S02` Absent clauses serialize as documented in Invariants
  (optional fields → `null`; ordered-list fields → empty array `[]`).
- `AC-393a-S03` A FROM item with no alias serializes its alias slot as
  `null` (not omitted, not empty string).
- `AC-393a-S04` Every new AST variant round-trips through serde
  `to_string` → `from_str` and compares equal.

### F — TS facade (AC-393a-Fc)

## Design Bar / Quality Bar

- Maintain the sprint-385 hand-written recursive-descent style. Do not
  introduce a parser-generator dependency (`nom`, `pest`, `logos`) — the
  WASM bundle budget makes this a hard constraint.
- No `unwrap()` / `expect()` on user-input paths. Lexer / parser errors
  return `Error(...)` variants. TS facade uses `unknown` + runtime guard.
- `parseSqlPreloaded` continues to be a pure synchronous function — no
