## Assumptions

- **Threshold mechanism**: chose the **`useDelayedFlag` hook** over inline-per-consumer because (a) the threshold logic is identical across all four surfaces, (b) the hook is independently testable with `vi.useFakeTimers` (cleaner than inline `setTimeout` audits), and (c) the synchronous-reset semantic on `active=false` is load-bearing for cancel→retry — encapsulating it in one place avoids drift.
- **Shared overlay location**: chose `src/components/feedback/AsyncProgressOverlay.tsx` because the `feedback/` directory is conventionally where ambient user-feedback components live (vs. `overlays/` which suggests modal layering). Decision recorded in `findings.md`.
- **Cancel-command name**: kept `cancel_query` (not renamed to `cancel_operation`). The wire signature is preserved per the invariant, and the cancel-token registry is already paradigm-neutral (it's just a `HashMap<String, CancellationToken>`). Renaming would be cosmetic-only and would invalidate the existing `cancel_query` tests + frontend consumers — net negative.
- **Cancel-registry shape**: kept the **single registry** at `src-tauri/src/commands/connection.rs:80` (`query_tokens: Mutex<HashMap<String, CancellationToken>>`). Per-paradigm registries would duplicate the cleanup pattern without semantic benefit.
- **Backend trait extension shape**: chose to add `Option<&'a CancellationToken>` (rather than required `&CancellationToken`) so legacy callers can pass `None` and remain bit-for-bit equivalent to pre-180. The cancel parameter is positioned last across all 8 trait methods for grep-ability. The PG impl converts `None` → no `tokio::select!` (direct execution); `Some(token)` → `tokio::select!` race that drops the in-flight future on cancel (client-side cooperative abort; no `pg_cancel_backend` SQL is issued at this sprint). The Mongo impl follows the same pattern.
- **`"cancelled"` history visual treatment**: chose `bg-muted-foreground` for the `QueryLog` dot and `bg-muted/40` for the `GlobalQueryLogPanel` row + `CircleSlash` icon (`lucide-react`). Both stay within the spec's "calm secondary, not destructive" tone. The icon distinction (`CheckCircle2` / `CircleSlash` / `XCircle`) keeps the three-way status visually scannable without color-only signal.
- **Real-timer waits in two refetch tests**: `DocumentDataGrid.refetch-overlay.test.tsx` and `rdb/DataGrid.test.tsx` use `findByRole("status", { name: "Loading" }, { timeout: 2000 })` instead of fake-timer `advanceTimersByTime`. Reason: `vi.useFakeTimers()` deadlocked the fetch microtask continuation when the document store's promise chain crossed `act` boundaries. Trade-off: ~6.6s extra wall-clock across the affected tests; acceptable for deterministic-pass rate.

## Residual Risk

- **Mongo cancel cooperative-only**: bundled mongo driver does not expose `Client::kill_operations`. Server-side query continues briefly after the user clicks Cancel; client drops the cursor future and never sees the result. UX matches PG exactly (overlay clears synchronously); only the server-side abort guarantee differs. Documented in ADR-0018 + operator runbook (`db.currentOp()` caveat).
- **SQLite Phase 9 forward-looking**: SQLite adapter is unimplemented. The trait signature carries `Option<&CancellationToken>` so the Phase 9 implementation can wire `sqlite3_interrupt(db_handle)` — or accept best-effort no-op — without breaking the frontend Cancel UX. Documented in ADR-0018.
- **250–999ms grey zone**: queries that resolve in 0.25–1.0s paint no overlay at all. Doherty 1s is the perception ceiling but mid-range responses still benefit from a lighter cursor / status hint. Not an AC violation (AC-180-01 explicitly says sub-1s ops do NOT show the affordance), but a UX iteration target.
- **Cancel→retry race against backend pool**: when server-side abort lands as a future enhancement (ADR-0018), racing a new fetch with cancel may surface a `query was cancelled` error from the previous query reaching the pool. Currently moot since cancel is client-side cooperative drop only. Deferred instrumented test target for the follow-on sprint that introduces server-side abort.
- **Pre-existing test failure** in `src/__tests__/window-lifecycle.ac141.test.tsx > AC-141-1 (real)`: unrelated to Sprint 180 (Sprint 175 lazy-workspace ADR-0017 stripped the `workspace` window from `tauri.conf.json`). Confirmed via `git diff HEAD -- src/__tests__/window-lifecycle.ac141.test.tsx` (empty). Not blocking Sprint 180 evidence.
- **Manual operator browser smoke (Verification Plan §Required Checks #9) NOT performed in this sandbox** because `pnpm tauri dev` requires a windowed environment. The frontend Cancel UX is fully exercised by the per-surface Vitest tests; the trait-level cancel cooperation is exercised by the 8 fake-adapter Rust unit tests. Live-DB PG `pg_sleep(5)` client-side drop verification and live-DB Mongo long-aggregate cooperative-drop verification (via `db.currentOp()`) are documented in `findings.md` § "Operator runbook" for operator replay. Note: server-side abort verification (`pg_stat_activity` shows pid removed) is deferred to the follow-up sprint that introduces `pg_cancel_backend(pid)` invocation.
- **Min window size 1024×600 visual confirmation NOT performed** for the same reason. Risk is low: `<AsyncProgressOverlay>` uses the same absolute-inset geometry as the pre-Sprint-180 inline overlays, which were already audited at min size in Sprint 176.
