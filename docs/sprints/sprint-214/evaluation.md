## Findings

- **F-001 [P3]**: Contract `§Verification Plan` check 5 baseline says "24 cases" but the actual pre-sprint test count for the 4 regression files is 26 (`ColumnsEditor.test.tsx` 18 + `IndexesEditor.test.tsx` 4 + `ConstraintsEditor.test.tsx` 2 + `SqlPreviewDialog.test.tsx` 2). Generator correctly reported 26; contract authoring was off by 2. Recommend updating future contracts to enumerate per-file test counts.

- **F-002 [P3]**: Contract `§Verification Plan` check 15 specifies the import path as `@components/structure/useDdlPreviewExecution`, but the project's `vite.config.ts` / `tsconfig.json` does not register an alias that resolves `@components/structure/<file>` for hooks-of-components. The 3 editors live alongside the hook and use the canonically correct relative path `./useDdlPreviewExecution`. Generator's choice is the only viable form. The grep wording in contract should be corrected to either `from "./useDdlPreviewExecution"` or `useDdlPreviewExecution` (selector-only) for future sprints.

- **F-003 [P3]**: `runCommit` `useCallback` deps include `previewSql`, so `attemptExecute` identity changes on every preview-set. No behavioral impact (regression tests pass), and `SqlPreviewDialog`'s `onConfirm` prop is invoked on click rather than identity-compared. Generator already flagged this in residual risk for a future cycle; could be optimized to `useRef<string>` for latest sql in a follow-up sprint.

- **F-004 [P3]**: Race condition on rapid double-click of "Review SQL" / "Preview SQL" / drop-button buttons carries forward from pre-sprint. Two `loadPreview` invocations can interleave; the second `pendingExecuteRef.current = prepareCommit()` clobbers the first. Generator explicitly noted this is out of scope for a refactor-only sprint. Recommend a follow-up to add a `requestId` or AbortController.

- **F-005 [P3]**: ColumnsEditor's `handleReviewSql` sets `setShowSqlModal(true)` *before* awaiting `ddl.loadPreview`. If `loadPreview` rejects, the modal is mounted with `previewError` set and empty `previewSql`. This is byte-equivalent to pre-sprint behavior (SqlPreviewDialog handles the empty/error branch), so it is a faithful refactor — but a future cycle could move the modal mount inside the success branch for symmetry with IndexesEditor / ConstraintsEditor (which both set `setShowPreviewModal(true)` before `loadPreview` as well, so the mount-then-fetch pattern is consistent across 3 editors). No P1/P2.

- **F-006 [P3]**: 9 grep hits for `useDdlPreviewExecution` across 3 editors include 3 doc comments (e.g. `ColumnsEditor.tsx:360`). Generator counted these in the 9 total. Contract's "≥ 6" only requires import + call (2 per editor), and that minimum is also met (6 hits even without the comments). No issue.
